Micro-Project — Assesment

Core Rule & System Clarity (20/20)

The system is organized by a singular rule: "A stone's kinetic skip-potential is determined by the vector of a user's drag-input until friction or boundary collisions force a state change." The slingshot mechanic consistently reflects this rule.

State Progression (15/15)

Justification: The system moves through four structural states:

    1. Potential (Stones on sand), 2. Dragging (Slingshot UI active), 3. Kinetic (Physics/Skipping loop), 4. Empty (UI prompt to reset).
The behavior shifts from manual input to automated physics to resource restriction.

Interaction & Evolution (15/15)

Justification: Interaction has meaningful consequences: pull distance scales skips from 1 to 8. The system degrades as the stone supply is exhausted and accumulates visual history through the persistent ripple array.

Iterative Development & Critique (15/15)

Justification: Based on peer feedback, I added a "Ghost Stone" at the origin point because users were confused about where to start the drag. I also refactored the physics to include a "sink" condition if the stone hits the pond's edge, because of the feedback i found out there was a bug solving a bug which randomly gave extra stone to skip

Documentation & Logging (15/15)

Justification: My code documents the implementation of the elliptical boundary formula "(x^2/a^2 + y^2/b^2 = 1)" to constrain the stones. I logged the transition from a simple "click-to-ripple" to a full skipping physics system.

Risk & Experimentation (8/10)

Justification: I pushed beyond basic reactive sketches by implementing State-Based Feedback. I mapped drag distance to a discrete `bounceLimit` array and used `drawingContext.clip()` to manage complex visual masking, ensuring ripples never bleed onto the "land" pixels. But I think the stone is a bit shaky because of the masks.

Participation & Critique Engagement (10/10)

Justification: Critique from my peers and instructor fundamentally shifted the system's structural priorities. Based on these critiques, I implemented three major changes to improve system clarity and stability:

  • Visual Feedback (The "Ghost Stone"): A peer mentioned they were confused and tried to drag stones directly from the land. To fix this, I added a "Ghost Stone" at the point of origin during the drag phase to clarify that the interaction is a slingshot mechanic starting from the pond.
  • Dynamic State Change: I implemented a visual feedback system on the slingshot vector. This allows the user to see the "charge" of the throw, communicating how many bounces (skips) the stone will achieve before the interaction is even released.
  • Structural Bug Fix: Peers identified a "teleportation" bug where a stone would reset prematurely if a new one was clicked too rapidly. I refactored the physics loop from a single-variable system into a Multi-Stone Array, allowing independent stone objects to exist and calculate physics without overwriting each other's data.
